GLM Component Reference

The source-of-truth reference for GLM components and styles. Every component shows real production markup you can lift into Razor. Styling comes from the shared foundation CSS.

OVERRIDDEN production customizes this via MDB remaps. DEFAULT renders on stock MDB5 (a candidate to review). GAP stock today but a Foundant treatment is drafted/needed. WIP section not yet rebuilt to real markup.

Colors OVERRIDDEN

The Foundant token palette. All values come from foundant-tokens.css.

WhatThe canonical color tokens. Everything in GLM references these, never raw hex.
Where--colors-* in foundant-tokens.css. Components consume them via --mdb-* remaps.
WhenBrand primary for text/primary actions. Neutrals for surfaces, borders, muted text. Semantic colors only for state (success/danger/warning/info).
WhyOne palette keeps the app coherent as it modernizes. Referencing tokens (not hex) means a palette change updates everywhere at once.
HowIn CSS: color: var(--colors-brand-primary-dark-blue). In markup, prefer utility classes (.text-muted, .text-danger) that already map to tokens.

Typography OVERRIDDEN

Roboto throughout. Compressed heading scale for a data-dense interface.

WhatThe heading scale and body text styles. One font family (Roboto), weight 500 for headings.
WhereReal h1–h6 elements, sized by the foundation CSS. Body copy is 1rem/400.
WhenUse the heading level that matches document structure, not visual size. Never pick a heading for its looks; if you need a small label, use .section-eyebrow (a non-heading), not h5.
WhyScale is compressed vs MDB defaults for density. Weight 500 (not bold) keeps headings calm. h3 is the same size as body and relies on weight for hierarchy.
HowJust use the tag: <h2>Section</h2>. Card titles use <h5 class="card-title">.

Icons OVERRIDDEN

Font Awesome Pro. Default family fa-light; fa-regular for filled-state controls.

WhatFont Awesome Pro icons. GLM uses the light family by default for a clean, thin look.
Where<i class="fa-light fa-*">. FA's JS upgrades it to inline SVG at runtime.
WhenUse fa-light as default. Use fa-regular where production does (toggles, the kebab menu). Size with FA classes (fa-lg) and space with Bootstrap utilities (me-2).
WhyOne icon family keeps weight consistent. Never mix in emoji or inline hand-drawn SVG.
How<i class="fa-light fa-plus me-2"></i>Add
This public reference CDN renders Font Awesome 6 Free, so icons here appear heavier than production's Pro fa-light. The class names shown are the correct production classes to lift; only the rendered weight differs.

Grid System DEFAULT

WhatThe 12-column responsive grid. Standard Bootstrap breakpoints and gutters.
Where.row + .col-*. Gutters via .g-* (record pages use .g-3).
WhenPair cards two-up on wide screens with .col-12.col-xl-6 so they stack on narrow ones. Use .col for equal-width auto columns.
WhyNo GLM override needed. Consistency comes from using the same column pattern across record pages, not from custom grid CSS.

Spacing & Radius OVERRIDDEN

Bootstrap spacing utilities. Custom Foundant radius scale.

WhatSpacing uses Bootstrap's m-*/p-* utilities. Radius uses the Foundant --radius-* scale.
WhereSpacing: utility classes in markup. Radius: --radius-* tokens in CSS.
WhenButtons/pills use round. Cards and the hero use card (6px). Inputs and small chips use sm (4px). Larger surfaces use md (8px).
WhyA fixed radius scale keeps corners consistent. --radius-card (6px) was added to name the value production hardcodes on cards, so prototypes stop using a magic number.
Howborder-radius: var(--radius-card). In markup, spacing like class="mt-2 mb-3".

Spacing utilities

Bootstrap scale: p-1 4px · p-2 8px · p-3 16px · p-4 24px · p-5 48px. Same values for margins (m-*) and directional variants (mt-, px-, etc.).

Buttons OVERRIDDEN

Styled by remapping --mdb-btn-* variables onto Foundant tokens. No bespoke button classes.

WhatStandard MDB buttons. The primary action control across GLM.
Where.btn base + variant. Page-level actions, card headers, modal footers.
WhenPrimary for the main affirmative action (one per view). Outline-dark is the record-page workhorse. Danger for destructive actions. Link for low-emphasis inline actions.
WhyBase .btn sets --mdb-btn-border-radius: var(--radius-round), --mdb-btn-font-size: 14px, text-transform: initial. Variants remap bg/hover/focus/active to tokens.
HowAdd a size (.btn-sm) and weight (.fw-bold) modifier as needed. Record-page default is .btn.btn-outline-dark.btn-sm.fw-bold.

Ratified variants

<button class="btn btn-primary">Primary</button> <button class="btn btn-dark btn-sm fw-bold shadow-none">Dark</button> <button class="btn btn-outline-dark btn-sm fw-bold">Outline dark</button> <button class="btn btn-outline-secondary">Outline secondary</button> <button class="btn btn-danger">Danger</button> <button class="btn btn-outline-danger">Outline danger</button> <button class="btn btn-link">Link</button>

Sizes

<button class="btn btn-primary btn-sm">Small</button> <button class="btn btn-primary">Default</button> <button class="btn btn-primary btn-lg">Large</button>

States

<button class="btn btn-primary">Normal</button> <button class="btn btn-primary" disabled>Disabled</button>

GAP — production Disabled uses MDB's built-in fade (opacity: 0.65) via the disabled attribute. This foundation CSS remaps --mdb-btn-disabled-bg to Foundant navy so it fades the right color; production has not yet added this remap, so a disabled button in the live app currently fades MDB's default blue instead. Fix: add the disabled remaps to mdb-customizations.css.

Cards OVERRIDDEN

Content containers. MDB .card with header/footer borders removed via customizations.

WhatThe primary content container. Groups related fields, actions, or data on a record page.
Where.card > .card-body. Record pages arrange them in a .row.g-3 grid.
WhenUse for any discrete content module. Pair two per row on wide viewports with .col-12.col-xl-6. Use .h-100 so paired cards match height.
WhyCustomizations remove .card-header bottom border and .card-footer top border for a flatter look. Card corners render at 6px (--radius-card).
HowCard header is a flex row: an <h5 class="card-title mb-0"> plus an actions cluster. Body holds the content.

Badges & Status OVERRIDDEN

Pill badges for status. Four semantic backgrounds; dark text on subtle tints.

WhatStatus indicators. Communicate the state of a record (stage, approval, urgency) at a glance.
Where.badge.rounded-pill + a semantic class. In hero cards, list rows, table cells.
WhenUse to show state, not to label fields. One badge per status dimension. Map each domain status to the semantic color that matches its meaning (see status map below).
WhyBase .badge sets text to --colors-brand-primary-dark-blue. The four badge-* classes use subtle -50 backgrounds with dark text, always readable.
HowAdd .fw-normal if the default weight is too heavy for the context.

Ratified semantic badges

Success Warning Danger Secondary
<span class="badge rounded-pill badge-success">Success</span> <span class="badge rounded-pill badge-warning">Warning</span> <span class="badge rounded-pill badge-danger">Danger</span> <span class="badge rounded-pill badge-secondary">Secondary</span>

Domain status map GAP — mapping not ratified

Approved Denied Pending LOI Draft Overdue

GLM's real statuses (Approved, Denied, Pending, In Review, LOI Draft, Overdue, Cancelled…) need a documented status-to-semantic mapping so the same status always gets the same color. The pairings shown are illustrative, not ratified. This mapping is a design + domain decision, pending.

Solid / white-text badges GAP — not in production

Primary Danger

Production has no solid, white-text badge variant. Shown inline here only. Do not lift; use the subtle semantic badges above.
