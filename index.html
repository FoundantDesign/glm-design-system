GLM Component Reference

The source-of-truth reference for GLM components and styles. Every component shows real production markup you can lift into Razor. Styling comes from the shared foundation CSS.

OVERRIDDEN production customizes this via MDB remaps. DEFAULT renders on stock MDB5 (a candidate to review). GAP stock today but a Foundant treatment is drafted/needed. WIP section not yet rebuilt to real markup.

Colors OVERRIDDEN

The Foundant token palette. All values come from foundant-tokens.css.

WhatThe canonical color tokens. Everything in GLM references these, never raw hex.
Where--colors-* in foundant-tokens.css. Components consume them via --mdb-* remaps.
WhenBrand primary for text/primary actions. Neutrals for surfaces, borders, muted text. Semantic colors only for state (success/danger/warning/info).
WhyOne palette keeps the app coherent as it modernizes. Referencing tokens (not hex) means a palette change updates everywhere at once.
HowIn CSS: color: var(--colors-brand-primary-dark-blue). In markup, prefer utility classes (.text-muted, .text-danger) that already map to tokens.

Typography OVERRIDDEN

Roboto throughout. Compressed heading scale for a data-dense interface.

WhatThe heading scale and body text styles. One font family (Roboto), weight 500 for headings.
WhereReal h1–h6 elements, sized by the foundation CSS. Body copy is 1rem/400.
WhenUse the heading level that matches document structure, not visual size. Never pick a heading for its looks; if you need a small label, use .section-eyebrow (a non-heading), not h5.
WhyScale is compressed vs MDB defaults for density. Weight 500 (not bold) keeps headings calm. h3 is the same size as body and relies on weight for hierarchy.
HowJust use the tag: <h2>Section</h2>. Card titles use <h5 class="card-title">.

Spacing & Radius OVERRIDDEN

Bootstrap spacing utilities. Custom Foundant radius scale.

WhatSpacing uses Bootstrap's m-*/p-* utilities. Radius uses the Foundant --radius-* scale.
WhereSpacing: utility classes in markup. Radius: --radius-* tokens in CSS.
WhenButtons/pills use round. Cards and the hero use card (6px). Inputs and small chips use sm (4px). Larger surfaces use md (8px).
WhyA fixed radius scale keeps corners consistent. --radius-card (6px) was added to name the value production hardcodes on cards, so prototypes stop using a magic number.
Howborder-radius: var(--radius-card). In markup, spacing like class="mt-2 mb-3".

Hover treatment GAP — resolution proposed, not yet in production

Production's top-level nav-link hover (.navbar-nav > li > a:hover) is a hardcoded gray gradient (#eee → #e4e4e4) with a 5px solid #222 border-bottom — the last hardcoded-hex hover left in the system. This prototype replaces it with background-color: var(--colors-state-layers-light-hover) and var(--radius-sm), matching the treatment already ratified for .kebab-btn, so the top-level toggle matches its own dropdown submenu instead of introducing a third hover language. Confirm with engineering before treating as ratified.

Separately: production's dropdown toggles pair MDB5's own .dropdown-toggle::after arrow with a leftover Bootstrap-3 <span class="caret">, and .caret has no rule anywhere in production CSS. Likely renders a duplicate indicator. Reproduced faithfully above since it's real production markup — flagging rather than silently dropping it.

Info card background GAP — inline style, not a class rule

.request-info-card carries no CSS of its own anywhere in production — its --colors-neutral-70 background is set with an inline style attribute directly in the Razor markup rather than a class rule. It works, but it means the color can drift silently wherever this markup is copy-pasted. Consider moving it into a real class rule (e.g. .request-info-card { background-color: var(--colors-neutral-70); }) the next time this page is touched.

Production badges are genuinely pill-shaped and uppercase — confirmed ratified in GLM.Summary.css, not a stock-MDB leftover like Alerts/Tabs. This is a deliberate departure: rectangular (--radius-sm) since pills read too similar to pill buttons, uppercase/letter-spacing removed, and a saturated border added per variant for contrast against white. Text stays uniform navy for now — colored text per variant is a separate, still-open decision (see Colors section proposed ramps). Needs engineering review before it round-trips into GLM.Summary.css.

WhatInline contextual feedback: success, warning, danger, info.

WhenPage-level or form-level feedback that isn't tied to a single field (field errors use .invalid-feedback, not this).
WhyMirrors the badge convention already ratified: a -50 tint background with --colors-brand-primary-dark-blue text, since only the danger ramp is WCAG-validated past -50 today (contract backlog #4).

Draft, not lifted from production evidence — no .alert customization exists today. Sets background-color/color directly in addition to the --mdb-alert-* custom properties, for the reason already established: MDB's stock .alert-info renders #e5f4f8/#3b7e94 (its own built-in subtle-info colors) regardless of the custom-property override. Border uses the full border: 1px solid ... shorthand rather than only border-color, since MDB's base .alert may not itself set a visible border-width — same defensive pattern as the color fix. Needs engineering review before it round-trips into mdb-customizations.css. Backlog item. alert-info uses -90/-50 rather than -50/-0 like its siblings — info-50 is noticeably more saturated than success-50/warning-50 and duplicates --colors-brand-secondary-light-blue (the sidenav active-state color); -90 matches the paleness of the other three.
